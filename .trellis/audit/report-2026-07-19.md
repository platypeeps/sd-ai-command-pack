## Findings

### Tooling

#### [A-035] P2 · M · Plausible — Post-review task lifecycle changes leave the generated knowledge base stale
- evidence:
  - `templates/.agents/skills/sd-ship/SKILL.md:93` — Stage 1 refreshes specs and generated knowledge before review.
  - `templates/.agents/skills/sd-ship/SKILL.md:114` — Stage 4 archives the task after that final refresh.
  - `scripts/sd-ai-command-pack-update-spec-kb.py:307` — active and archived task Markdown is generated knowledge input.
  - `templates/scripts/sd-ai-command-pack-full-check.sh:517` — stale generated knowledge fails full-check when the local package exists.
  - `make check` — failed only on task-copy/dashboard/overview freshness after all earlier lanes passed.
- why: A normal ship can leave clean `main` unable to pass its canonical local gate on a machine that maintains `.obsidian-kb`.
- fix: Add a final conditional KB refresh owner after task archive and post-merge follow-ups, with an end-to-end freshness regression.

### Documentation

#### [A-036] P2 · S · Plausible — Removed review-body fallback remains documented in the shipped full-check skill
- evidence:
  - `templates/.agents/skills/sd-full-check/SKILL.md:110` — advertises `REVIEW_PREFLIGHT_PR_BODY` through `0.15.x`.
  - `CHANGELOG.md:220` — records removal of that fallback in `0.16.0`.
  - `tests/test_review_scope.py:248` — retirement coverage checks installed guides but not shipped skills.
- why: A consumer following the skill can configure a no-op variable and still fail PR-body scope resolution.
- fix: Remove the stale entry and extend the retired-surface regression across all shipped docs and skills.

  - `.trellis/tasks/07-19-status-read-only-bytecode/task.json:6` — status is completed outside the archive.

### Security

#### [A-027] P3 · S · Plausible — Trellis lifecycle hooks execute config-sourced commands via shell=True (upstream-owned)
- evidence:
  - `.trellis/scripts/common/task_utils.py:260` — iterates repository-configured hook commands.
  - `.trellis/scripts/common/task_utils.py:262` — invokes each hook through `subprocess.run`.
  - `.trellis/scripts/common/task_utils.py:264` — explicitly enables `shell=True`.
  - `.trellis/scripts/common/config.py:285` — loads hook configuration from the repository.
- why: The behavior matches trusted-repository hook semantics but the upstream contract and consent boundary remain implicit.
- fix: Keep the pack-side task parked; upstream Trellis should document shell semantics, add consent, or support argv lists.
